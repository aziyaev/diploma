\begin{abstract}
В данной работе описаны результаты, достигнутые в ходе расширения функциональности системы дистанционного обучения «МГУ Контест», усовершенствования процесса проверки работ студентов и обеспечения безопасности при компиляции и запуске пользовательских программ в системе.

Произведено перепроектирование существующей программной структуры системы для реализации в системе новых видов выдаваемых для решения студентами задач. Внесены структурные решения и описан пользовательский интерфейс, реализующие новые виды выдаваемых задач и способы взаимодействия с ними пользователей. Применён инструмент для изоляции процесса компиляции и запуска пользовательских программ в системе с целью обеспечения безопасности на сервере. Внедрено решение для редактирования пользовательского текста.
\end{abstract}

\selectlanguage{english}
\begin{abstract}
This paper describes the results achieved in the course of expanding the functionality of the distance learning system "MSU Contest", improving the process of checking students' work and ensuring safety when compiling and running user programs in the system.

The existing software structure of the system was redesigned for the implementation in the system of new types of tasks issued for solving by students. Structural solutions are introduced and a user interface is described that implements new types of issued tasks and how users interact with them. A tool has been used to isolate the compilation process and launch of user programs in the system in order to ensure security on the server. Implemented a custom text editing solution.
\end{abstract}

\section{Реализация новых типов выдаваемых для решения студентами задач}

В данной части работы описываются основные изменения и результаты реализации следующих новых типов выдаваемых для решения студентами задач:

\begin{itemize}
    \setlength\itemsep{-0.2em}
    \item задача с возможностью указать варианты ответа;
    \item задача с возможностью предоставить текстовый ответ;
    \item задача с возможностью загрузить файлы в качестве ответа.
\end{itemize}

Реализация новых типов выдаваемых (для решения студентами) задач, схожих по общему смыслу с существующим единственным видом задач, но отличным в деталях, может потребовать предварительного внесения правок и добавления новых свойств и механизмов в существующую структуру. В частности, \emph{сущность, представляющая собой выдаваемую студентам задачу}, а также \emph{сущность, представляющая решение задачи}, и интерфейс взаимодействия с этими двумя сущностями претерпевают изменения в сторону расширения количества включаемых атрибутов и механизмов взаимодействия с ними.

Реализация типа задачи подразумевает также реализацию пользовательского интерфейса для взаимодействия пользователей различных ролей с конкретным типом задач: создание и редактирование задач, решение задачи и решение задачи по назначению преподавателем, проверка решения задачи.

Теперь задача включает обязательные атрибуты с информацией о критериях оценивания в баллах. От пользователя требуется ввести максимальный балл, который можно получить за решение задачи, и критерии (необходимое количество баллов) для получения оценки 3, 4 и 5.

Реализован \emph{тип задачи «Варианты ответа»} с возможностью при создании указать возможные варианты ответа и отметить правильные. Проверка решения такой задачи осуществляется автоматически, выносится вердикт и выставляется оценка. При решении задачи студент получает форму, которая позволяет выбрать один или несколько вариантов в зависимости от количества действительно правильных ответов, указанных при создании задачи.

Реализован \emph{тип задачи «Текст»}, в качестве решения которой принимается текст, набранный студентом в поле на веб-странице решения задачи. Задачи данного типа проверяются преподавателем.

Аналогичный тип задачи «Файлы» подразумевает решение в виде прикреплённых файлов определённого типа.

Также для задачи реализована возможность \emph{содержать набор подзадач} (это задача типа «Тест»). Для создания такой задачи пользователь при создании задачи (преподаватель или администратор системы) выбирает существующие (ранее созданные) задачи, которые требуется включить в качестве подзадач. Таким образом новая функциональность позволяет агрегировать несколько задач в набор. Важно отметить, что такой набор не может быть включен как подзадача.

При создании теста критерии оценивания указываются не в баллах, а в процентах. За 100\% считается сумма максимальных баллов всех подзадач. При проверке суммируются все набранные баллы по подзадачам и вычисляется процент по отношению к сумме максимальных баллов всех подзадач. Это и есть значение (процент) правильности решения теста.

Такой тест может выдаваться в качестве задания для студента. Студент, начав решать тест, поочерёдно получает для решения задачи из теста.

\section{Изоляция процесса компиляции и запуска пользовательских программ}

Изначально имеющийся в системе тип задач «Программа» поддерживает автоматическую проверку решений. Для решения задач этого типа студент загружает набор исходных файлов программы. После загрузки файлов студентом в системе инициируется компиляция исходных файлов и запуск полученного в результате компиляции исполняемого файла.

Данный процесс может быть небезопасным для операционной системы на сервере, где осуществляется его запуск, например, в случаях, когда студент неумышленно допускает ошибки в реализации сдаваемой программы или когда студент умышленно реализует вредоносную программу.

Одним из способов изоляции этого процесса является его запуск на отдельной виртуальной машине. Но многократное создание и параллельное поддержание нескольких виртуальных машин является ресурсозатратным для физической машины. Другой способ, который и был выбран для применение, --- это \emph{механизм контейнеризации}.

Контейнеризация --- это парадигма операционной системы, в которой ядро допускает существование нескольких изолированных экземпляров пользовательского пространства. Такие экземпляры, называемые \emph{контейнерами}, могут выглядеть как настоящий компьютер с точки зрения запущенных в нём программ. Компьютерная программа, работающая в обычной операционной системе, может видеть все ресурсы (подключенные устройства, файлы и папки, общие сетевые ресурсы, мощность процессора, поддающиеся количественной оценке аппаратные возможности) этого компьютера. Однако программы, работающие внутри контейнера, могут видеть только его содержимое и устройства, назначенные контейнеру.

Docker --- это набор программных продуктов, которые используют виртуализацию на уровне операционной системы для доставки программного обеспечения в контейнерах. Контейнеры изолированы друг от друга и объединяют собственное программное обеспечение, библиотеки и файлы конфигурации; они могут общаться друг с другом через четко определенные каналы. Поскольку все контейнеры Docker совместно используют службы одного ядра операционной системы, они используют меньше ресурсов, чем виртуальные машины.

\section{Форма редактирования текста}

Реализованная форма редактирования пользовательского текста предоставляет обширную функциональность для редактирования и визуального форматирования текста в поле для ввода на веб-странице, а также возможность копирования из Microsoft Word с сохранением исходного форматирования. Наличие такой формы означает для пользователей работу в среде с привычными инструментами редактирования и форматирования текста и возможность просматривать сохранённый форматированный текст на веб-странице без необходимости выгружать и загружать файл с текстом для обеспечения доступа к тексту других пользователей.

Для внедрения такой формы в систему «МГУ Контест» используется пакет «Django CKEditor» \cite{ckeditor} для фрейморка Django, предоставляемый в качестве готового проверенного решения в каталоге пакетов языка программирования Python.
